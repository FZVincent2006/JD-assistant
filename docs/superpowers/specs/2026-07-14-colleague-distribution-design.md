# 招聘 JD 发布助手同事分发设计

## 目标

为同一飞书租户内的四位实际使用者提供一个无需服务器、同时兼容 macOS Intel 与 Apple Silicon、Chrome 与 Edge 的可分发安装包。四台电脑共用现有企业自建应用 `招聘 JD 发布助手`，但每位使用者仍以自己的飞书账号授权，并且只能写入固定测试副本。

同事姓名只在飞书管理界面中用于搜索用户和核对权限，不写入代码、Git、构建产物、压缩包、日志或安装脚本。

## 方案选择

采用“固定扩展 ID + 未打包扩展 + Universal 本机授权助手”的方案。

- 在扩展 manifest 中加入固定公钥，使同一构建在不同目录、电脑及 Chrome/Edge 中得到同一个扩展 ID。
- 生成公钥时使用的私钥不提交、不分发，也不写入最终压缩包；当前分发方式只需要 manifest 中的公钥。
- 飞书应用只需新增一个 OAuth 回调地址：由实现阶段生成的固定扩展 ID 加上标准的 `.chromiumapp.org/feishu` 回调路径；该完整地址会记录在版本文件中。
- 不采用 Chrome Web Store 或 Edge Add-ons，以免引入商店审核与发布维护。
- 不继续使用各机器随机扩展 ID，以免为每台电脑和浏览器重复添加回调地址。

固定 ID 会替代当前开发机的临时 Edge ID。旧回调地址在新包完成所有者验收前保留，确保可以立即回退；验收后再决定是否删除。

## 分发包内容

最终压缩包包含：

```text
招聘JD发布助手-macOS-20260714/
├── 扩展/
├── 原生助手/Feishu JD Assistant Helper.app
├── scripts/install-feishu-auth-helper.sh
├── 安装飞书授权助手.command
├── 安装说明.md
├── VERSION.txt
└── SHA256SUMS.txt
```

- `扩展/` 是经过生产校验的 `dist` 内容，可在 Chrome/Edge 的开发者模式中“加载已解压的扩展程序”。
- Universal 助手同时包含 `arm64` 与 `x86_64`，使用 ad-hoc codesign，安装到当前用户的 Application Support，不需要管理员权限或常驻服务。
- 双击 `.command` 安装入口使用包内固定扩展 ID 调用现有安装器，并在终端中隐藏读取 App Secret。
- App Secret 只写入当前用户的 macOS Keychain，不进入包或 shell 历史。建议由应用管理员在每台 Mac 上现场输入，不通过聊天发送给同事。
- 安装包不申请 macOS 辅助功能、屏幕录制、输入监控或完全磁盘访问。

如果 macOS 因内部传输产生的 quarantine 标记阻止首次运行，安装说明只引导用户使用系统提供的“隐私与安全性 → 仍要打开”；安装脚本不自动绕过 Gatekeeper。

## 飞书应用配置

沿用 App ID `cli_aade4224b8789bef` 和现有三个用户身份权限：

- `wiki:wiki:readonly`
- `docx:document:readonly`
- `docx:document:write_only`

配置顺序：

1. 在安全测试得到固定扩展 ID 后，将对应 OAuth 回调地址加入应用配置，保留当前回调地址。
2. 将应用可用范围限制为所有者和对话中明确提供的三位同事；不开放到全公司或整个部门。
3. 发布包含新回调与可用范围的应用版本。
4. 在固定测试副本文档中核对同一批用户的编辑权限，缺失时只添加缺失用户。
5. 不修改正式招聘文档的权限或内容。

飞书后台若要求管理员确认、扫码、二次认证或最终发布确认，由所有者在当前会话中完成该确认；自动化不会绕过身份验证。

## 安装与使用流程

每台 Mac：

1. 解压分发包，在 Chrome 或 Edge 开启扩展开发者模式，加载 `扩展/`。
2. 核对浏览器显示的扩展 ID 与 `VERSION.txt` 中的固定 ID 完全一致。
3. 双击 `安装飞书授权助手.command`，由管理员在本机输入 App Secret。
4. 完全退出并重新打开浏览器，在扩展中点击“授权飞书”。
5. 用自己的飞书账号完成授权，点击“检查测试副本”进行只读验证。
6. 首台所有者电脑通过后，再分发给另外三位同事。

公司 Heading 1 的 `1.` 自动序号继续由使用者写入成功后在飞书页面手动设置；分发版本不注入飞书页面、不滚动文档、不发送键盘快捷键。

## 安全与回退

- 正式文档没有写入入口，固定测试副本校验保持不变。
- Boss/脉脉受保护文件必须通过基线哈希校验。
- 包内扫描不得出现 App Secret、access token、refresh token、人员姓名或私钥文件。
- 新固定 ID 验收前保留旧扩展目录、旧 Native Messaging origin 和旧飞书回调地址。
- 若新包授权失败，可重新加载旧扩展并继续使用旧回调；不修改或撤销已存在的文档内容。

## 验证标准

打包前必须完成：

- JavaScript 全量测试、Boss/脉脉基线校验和生产构建全部通过。
- Swift 测试通过；Universal 助手同时验证 `arm64` 与 `x86_64`；app bundle codesign 验证通过。
- Chrome 与 Edge 读取同一 manifest 后计算出的扩展 ID 与发布记录一致。
- 最终压缩包解压后，扩展构建验证、shell 语法检查、应用签名检查和架构检查全部通过。
- 包内敏感信息扫描通过，`SHA256SUMS.txt` 与最终文件一致。
- 所有者先完成一次授权、检查测试副本和计划生成；不在打包验收阶段写入正式文档。
