# 飞书公司标题页面自动编号设计

## 背景与结论

测试副本已经证明：通过“创建子块”接口创建 Heading 1 时携带 `sequence: "auto"` 不会产生自动编号；创建后再调用“更新块”接口设置同一字段也会被飞书拒绝。结果是岗位 JD 内容已写入，但公司标题没有自动编号，Portfolio 阶段按安全规则停止。

本次改为混合写入：OpenAPI 继续负责所有结构化内容创建、文档读取和最终校验；扩展页面脚本只负责对刚创建的公司 Heading 1 执行一次飞书原生快捷键 `Command + Shift + 7`。OpenAPI 回读确认编号生效后，写入流程才可继续到 Portfolio。

## 范围

### 包含

- 新公司写入时，为岗位 JD 区的公司 Heading 1 开启飞书自动编号。
- 写入期间要求指定测试副本处于当前浏览器窗口的活动标签页。
- 为 `zhenfund.feishu.cn/wiki/*` 增加页面脚本运行权限，但运行时仍只允许固定测试副本 URL。
- 在侧栏写入结果中显示安全的错误码、HTTP 状态、Log ID 和失败阶段。
- 保留 OpenAPI 的写入顺序和最终结构校验。

### 不包含

- 不开放正式文档写入。
- 不使用 `debugger` 权限、系统剪贴板或未公开的飞书内部接口。
- 不使用 macOS 辅助功能、AppleScript 或本机助手发送键盘事件。
- 不自动修复此前遗留的“只有 JD、没有 Portfolio”的半成品公司块。
- 不修改 Boss/脉脉解析、字段映射、页面填写和诊断逻辑。

## 架构与数据流

### 1. 写入编排

后台写入器按以下顺序执行新公司计划：

1. OpenAPI 预检文档并生成完整写入计划。
2. OpenAPI 创建岗位 JD 公司块。
3. OpenAPI 回读，校验公司、介绍、岗位标题和引用块均已正确创建；此时暂不要求公司标题已经编号。
4. 后台调用独立的页面编号服务，传入公司名和固定测试副本 URL。
5. 页面编号服务检查活动标签页、定位唯一公司标题、执行一次快捷键并返回页面级结果。
6. 后台进行只读 OpenAPI 轮询，直到公司 Heading 1 的 `sequence` 为 `auto`，或达到超时上限。
7. 编号确认后将 JD 标记为完成，使用最新文档版本写入 Portfolio。
8. OpenAPI 回读并校验 Portfolio，成功后返回完整成功状态。

老公司追加岗位不创建新的公司 Heading 1，因此跳过页面编号步骤，保持现有 OpenAPI 流程。

### 2. 页面编号服务

页面编号服务由 Manifest 注册的内容脚本承载；后台只向当前活动的飞书测试副本标签页发送消息。它必须同时满足以下条件：

- 活动标签页 URL 规范化后与固定测试副本完全一致。
- 页面处于可编辑状态。
- 在虚拟滚动文档中只找到一个文本与公司名完全匹配的根级 `docx-heading1-block`。
- 该 Heading 1 当前没有 `.heading-order`，防止重复切换导致取消编号。
- 匹配块不是“岗位JD整理”“开放岗位”等分区标题。

定位过程可以滚动飞书文档，但不得编辑其他块。找到目标后：

1. 将目标滚动到可见区域。
2. 聚焦其 `contenteditable` 编辑区并将折叠光标放入标题。
3. 向目标编辑区发送一次 `keydown`/`keyup` 组合：`metaKey=true`、`shiftKey=true`、`key="7"`、`code="Digit7"`。
4. 在页面内短时等待 `.heading-order` 出现。只把它作为页面级信号；最终成功仍以 OpenAPI 回读为准。

快捷键不自动重试。页面找不到目标、目标不唯一、不可编辑、快捷键未产生页面编号控件时立即失败。

### 3. 权限与消息边界

- Manifest 只增加 `https://zhenfund.feishu.cn/wiki/*` 的 host permission 和 content-script match。
- 内容脚本收到新的 `FEISHU_APPLY_HEADING_NUMBERING` 消息时，必须再次检查固定测试副本 URL。
- Boss/脉脉原消息入口和页面匹配规则保持不变。
- 后台页面编号服务只查询当前窗口的活动标签页，不搜索、切换或修改其他标签页。

## 同步与校验

页面操作后，后台进行有限次数的只读检查。轮询只读取文档，不重复发送快捷键，也不重复创建块。每次使用飞书返回的最新文档版本。

成功条件：唯一匹配公司的根级 Heading 1 回读到 `headingSequence === "auto"`，且岗位 JD 其他结构仍通过完整校验。

失败条件包括：

- 活动标签页不是固定测试副本。
- 页面脚本不可用或页面不可编辑。
- 目标 Heading 1 缺失、重复或已经处于不可判定状态。
- 页面快捷键被编辑器忽略。
- 页面已显示编号，但 OpenAPI 在超时前始终没有回读到 `sequence: "auto"`。
- 编号后岗位 JD 其他结构发生变化。

所有失败都停止 Portfolio 写入，不撤销已创建的 JD，也不自动重复提交。

## 结果与诊断

新增或细化以下阶段：

- `jd-numbering-page`：活动标签页检查、定位或页面快捷键失败。
- `jd-numbering-verify`：页面操作完成，但 OpenAPI 未确认自动编号。

侧栏在写入状态后追加非空诊断信息：失败阶段、错误码、HTTP 状态和 Log ID。页面错误使用固定中文原因，不回传 DOM、文档正文、令牌或内部异常堆栈。

## 测试

### 单元和集成测试

- 精确定位唯一、根级、未编号的公司 Heading 1。
- 拒绝错误 URL、不可编辑页面、缺失目标、重复目标和已编号目标。
- 验证只发送一次 `Command + Shift + 7` 事件组合。
- 验证虚拟滚动定位和页面编号控件等待。
- 验证后台只对活动测试副本标签页发送编号消息。
- 新公司流程：JD 创建、页面编号、OpenAPI 轮询确认、Portfolio 写入。
- 页面编号失败或 OpenAPI 超时：Portfolio 不写入，状态和诊断正确。
- 老公司追加流程跳过编号步骤。
- Manifest 仅增加预期飞书文档权限。
- Boss/脉脉原有测试和完整性校验继续通过。

### 人工验收

测试前删除上次失败遗留的整段 CoFANCY JD 半成品，确保两个区域不存在该公司。

1. 在活动的测试副本标签页写入含两个岗位的新公司。
2. 确认岗位 JD 公司标题自动显示 `1.`，后续公司自动重排。
3. 确认公司介绍、岗位引用块和 Portfolio 均在正确位置。
4. 再追加一个新岗位，确认不创建第二个公司标题且不再次触发编号。

如果页面脚本生成的快捷键无法驱动飞书编辑器，则本方案判定不可行；停止继续增加 DOM 绕过手段，另行评估需要 macOS 辅助功能授权的本机真实键盘方案。
