# 飞书文档操作界面简化设计

## 目标

降低首次使用者的决策成本，让正常流程只呈现当前阶段需要执行的主要动作，同时完整保留 OpenAPI 写入前后的权限、模板、重复岗位、文档版本与结构校验。

## 方案比较

1. 保留现状并补充说明：改动最小，但使用者仍需判断“打开文档检查”和“检查正式招聘文档”是否必点，不能解决主要困惑。
2. 隐藏手动检查入口并复用现有自动预检：正常流程最短，且“生成写入计划”本来就会读取最新文档并生成基于 revision 的计划。采用此方案。
3. 合并计划与写入为一个按钮：步骤最少，但会削弱写入前预览计划和人工确认，不适合正式招聘文档。

## 正常流程

飞书文档模式的正常操作顺序调整为：

1. 未授权时显示“授权飞书”；授权成功后显示简洁的“飞书已授权”状态。
2. 使用者粘贴语料并点击现有解析按钮，检查可编辑预览。
3. 点击“检查并生成写入计划”。该动作继续通过 `FEISHU_PLAN` 读取最新正式文档，完成权限、模板、公司、岗位、重复项和 revision 检查。
4. 计划有效时显示现有计划摘要，并允许点击“确认写入正式文档”。
5. 写入继续执行 JD 写入、JD 回读校验、Portfolio 写入和 Portfolio 回读校验。

不合并“生成计划”和“确认写入”，以保留计划预览与最后一次系统确认。

## 界面变化

- 从常驻访问面板移除“打开文档检查”。
- 从常驻访问面板移除“检查正式招聘文档”。
- 已授权时不再把“重新授权”作为主要按钮；提供弱化的文字入口，供令牌或账号异常时使用。
- 将计划按钮文案从“生成写入计划”调整为“检查并生成写入计划”，明确检查已经内置。
- 正常页面仍显示固定目标为正式招聘文档以及授权状态，但减少说明文字和按钮密度。
- 仅当写入结果为部分完成、结果未知或校验失败，并且人工查看能帮助恢复时，显示“打开正式文档检查”入口。
- 错误文案中继续保留具体诊断和人工修复提示，不自动重试或撤销。

## 状态与数据流

- `FEISHU_AUTHORIZE` 与现有 Keychain/原生助手流程不变。
- 删除独立 `FEISHU_INSPECT` 的日常 UI 触发，不删除后台检查能力，以免破坏既有接口和测试工具。
- `FEISHU_PLAN` 仍返回最新 inspection 和 write plan，计划与当前 revision 不一致时禁止写入。
- 编辑任何预览字段仍会使计划失效，必须再次点击“检查并生成写入计划”。
- 写入结果正常成功时不显示人工打开文档入口；异常结果按现有阶段信息决定是否显示。

## 兼容性边界

- 不修改 Boss/脉脉解析、字段映射或页面填充逻辑。
- 不修改飞书解析器、OpenAPI 写入顺序、结构渲染或回读校验标准。
- 正式文档 URL、固定扩展 ID、授权回调和安装流程保持不变。

## 测试与验收

- UI 测试确认常驻页面不再包含“打开文档检查”和“检查正式招聘文档”。
- UI 测试确认计划按钮显示“检查并生成写入计划”。
- UI 测试确认未授权时仍可授权，已授权时仍有弱化的重新授权入口。
- UI/格式化测试确认异常结果会提供正式文档人工检查入口，成功结果不会提供。
- 现有飞书计划、写入与回读测试全部继续通过。
- Boss/脉脉保护性基线校验、完整测试和生产构建继续通过。
