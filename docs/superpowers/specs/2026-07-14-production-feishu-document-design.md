# 飞书正式招聘文档切换设计

## 目标

将招聘 JD 发布助手从已验收的测试副本完全切换到唯一正式文档：

`https://zhenfund.feishu.cn/wiki/RTWjwVZjri4uCUk0J8wcn2K3n6d`

扩展不再提供测试副本入口或环境切换开关。Boss/脉脉现有解析与页面填充逻辑保持不变。

## 方案

### 唯一文档边界

- 把飞书配置中的测试语义常量改成正式文档语义常量，固定 Wiki token 为 `RTWjwVZjri4uCUk0J8wcn2K3n6d`。
- URL 校验继续只比较协议、域名和路径；允许正式 URL 自带查询参数或锚点，但拒绝任何其他 Wiki 文档。
- OpenAPI 检查、计划、写入、回读校验和旧页面辅助模块全部引用同一个正式目标常量。
- 构建验证必须证明正式 token 存在、测试 token 不存在，避免旧地址被打进安装包。

### 操作界面

- 环境标识改为“固定目标：正式招聘文档”。
- 文档卡片和链接直接展示正式文档 URL。
- 按钮改为“检查正式招聘文档”和“确认并写入正式招聘文档”；恢复入口改为“确认并恢复正式招聘文档”。
- 授权、检查、计划、写入、部分成功、结果未知和修复提示全部去掉“测试副本”措辞。
- 最终写入确认框明确显示“仅写入正式招聘文档”和公司/岗位数量，保留人工确认，不增加一键绕过方式。

### 数据流与失败策略

数据流保持已验收顺序：授权 → 只读检查 → 生成计划 → 人工确认 → 写入 JD → API 回读校验 → 写入 Portfolio → API 回读校验。

- 正式文档标题或模板不唯一、权限不足、revision 变化、重复公司/岗位或回读不一致时立即停止。
- API 结果未知时不自动重试；提示用户先人工检查正式文档，避免重复写入。
- 切换只改变唯一目标和界面语义，不放宽任何模板、重复检测或分阶段校验规则。

## 发布与迁移

- 扩展版本提升到 `0.2.1`，发布新的预发布标签 `v0.2.1-codex.1`，不复用或覆盖测试目标资产。
- 已发布但尚未激活的 `v0.2.0-codex.1` 在 Release 说明中标记为废弃测试版，不提供给同事。
- 最终 `distribution/release-channel.json` 只指向新的正式目标 ZIP、构建提交和权威 SHA-256。
- README、Codex 安装协议和安装包说明统一改为正式招聘文档，不保留测试 URL。

## 验证

- 先写失败测试，覆盖正式 URL 白名单、旧 URL 拒绝、UI 按钮/提示、构建产物中正式 token 存在且测试 token 缺失。
- 运行全部 Vitest、Boss/脉脉受保护文件哈希、生产构建、原生助手 64 项测试、arm64/x86_64 和签名校验。
- 发布前执行正式文档只读检查；只有唯一识别 Portfolio 与岗位 JD 两个区域且用户具有所需权限时才进入实际写入。
- 不使用示例公司对正式文档做额外测试写入；首个真实语料仍由扩展生成计划并经用户确认后写入。
